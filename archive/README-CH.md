
# 望见陆地：将你的代码仓库打造为智能体工作空间

> **AAMS — 自主智能体清单规范**  
> `README.md` 是给人类的。`AGENT.json` 是给机器的。

在茫茫大海上生存，需要两样东西：一支好的船员——以及一张人人都能读懂的地图。新水手能读。夜班接替者能读。AI 也能读。

这正是 AAMS —— 自主智能体清单规范。

## 问题依然存在。解决方案现在更清晰了。

第 48 个会话不知道第 47 个会话决定了什么。这是以前的问题。这也是今天的问题。世界上没有任何智能体框架能解决它——因为问题不在框架里。问题在代码仓库的结构里。

没有智能体结构的代码仓库，就像一艘没有航海日志的船。每个人都知道自己昨天做了什么。没有人知道在他之前发生了什么。

## 一年后的认知

我在真实项目中与 AI 智能体合作超过一年。我学到的是：

> 最重要的不是智能体能写代码。  
> 最重要的是它们知道**自己在哪里**。

没有结构，就会失去上下文。失去上下文，就会出错。重复决策。孤立文件。没有人订购的技术债务。

解决方案不是一个新的 AI。解决方案是**代码仓库中的纪律**。

---

## AAMS 不是框架

这是最重要的说明。

AAMS 不是工具。不是运行时。不是需要安装的框架。

AAMS 是一个**单一文件**，放入任何代码仓库：

```
.agent.json
```

读取这个文件的智能体立刻知道：

- 文档应该放在哪里
- 会话如何结构化
- 长期记忆在哪里
- 它被允许做什么——以及不被允许做什么

不需要 `npm install`。不需要 `pip install`。无需任何设置。

---

## 三层文档模型

这是 AAMS 的真正核心。三个层级，强制执行：

**工作文档（Workpaper）** — 我在这个会话中正在做什么？  
在会话开始时创建，在会话结束时归档。包含完整的文件协议：创建、修改、删除了什么。

**白皮书（Whitepaper）** — 这个系统是如何构建的？  
稳定的架构真相。只写一次，只在架构决策时更新。永不删除。

**长期记忆（Long-Term Memory）** — 我们随时间学到了什么？  
每次会话后，工作文档被摄入 LTM。第 N+1 个会话在开始前查询 LTM。不再有上下文丢失。

```
WORKING/
├── WHITEPAPER/     ← 稳定的系统真相。永不删除。
├── WORKPAPER/      ← 活跃的会话工作。每个会话一个文件。
│   └── closed/     ← 已归档的完成会话。
└── MEMORY/         ← 长期记忆。跨会话上下文。
```

一个优秀的开发者在脑海中完成这些。智能体需要显式且持久地完成这些。

## 智能体契约

代码仓库根目录下的 `READ-AGENT.md` 为每个进入该仓库的智能体定义了规范性契约。

任何以智能体身份进入此仓库的实体，都执行这个契约。无需讨论。无需解释。

```
首次进入：       读取 READ-AGENT.md → 检查结构 → 扫描仓库 → 索引到 MEMORY/
每次会话开始：   读取 READ-AGENT.md → 检查最后的工作文档 → 查询 MEMORY/
每次会话结束：   完成工作文档 → 摄入 → 移至 closed/ → 更新 READ-AGENT.md
```

适用于任何智能体框架。也适用于没有任何框架的情况。

## 可移植到任何代码仓库

核心设计目标：**一个文件，每个仓库**。

开发者有自己的智能体框架？他的框架识别 `WORKING/` 结构，并直接将其用作子智能体工作空间。

开发者没有框架？`.agent.json` + `READ-AGENT.md` 是最小可能的智能体框架——声明式、幂等、零依赖。

长期目标：AAMS 成为每个智能体在每个代码仓库中都能识别的事实标准。
